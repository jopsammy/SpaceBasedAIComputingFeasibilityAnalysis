## 二、项目目标与总原则

- 本项目的核心任务是通过调研、外部搜证与本地 Python 建模，尽可能逼近“太空算力真实情况”，而不是生产看似完整但证据链虚弱的叙事。
- 一切判据以真实资料、真实数据、可复核公式链、可追溯脚本结果为基础；假设只能作为最后一档补位，不得反客为主。
- 只要某个问题存在直接物理推导链、工程约束链或可本地建模核算路径，该路径就是第一优先级，强于任何二手转述。
- 外部资料不是为了替代本地核算，而是用于提供输入边界、权威锚点、交叉校验和迁移支撑。
- 计划阶段、锚点写作阶段、审计汇总阶段必须保持客观审计视角，禁止主观押注、倾向性裁定和先入为主的方向性结论。

## 三、证据等级体系

### 3.1 四档证据强度

1. **第一档：物理推导链 / 本地独立核算**
   - 适用条件：存在明确物理规律、工程约束、公式链或可本地 Python 独立建模的问题。
   - 强制要求：必须给出公式、关键输入、单位、脚本或明确的手工推导链，并以本地核算结果作为判断主依据。
   - 地位：全项目最高优先级证据。

2. **第二档：权威资料直接支撑**
   - 适用条件：不存在直接物理推导链，或推导链仍需要外部权威输入锁定边界。
   - 强制要求：必须给出明确来源 URL，并说明其权威性、时间、适用范围与为什么可以直接用于当前问题。
   - 示例范围：官方页面、监管/研究机构文档、正式论文、厂商规格页、正式财报/招股书/FCC 文件等。

3. **第三档：可迁移证据**
   - 适用条件：资料存在但不够权威，或虽权威但不能直接迁移，只能做类比/迁移使用。
   - 强制要求：必须给出来源 URL，并同时写清：
     - 为什么可以迁移
     - 可迁移的是哪一部分
     - 不可迁移的是哪一部分
     - 乐观 / 基准 / 保守三档预估依据分别是什么
   - 未满足上述四项时，不得采信。

4. **第四档：合理推断**
   - 适用条件：确实没有资料支撑，也没有更强替代路径，只能为了保持模型闭环做最弱补位。
   - 强制要求：必须明确写出推断理由、约束边界、为何无法获得更强证据、以及该推断只在哪个局部问题中暂时使用。
   - 地位：最弱证据；不得伪装成事实，不得直接升级为主结论。

### 3.2 现有项目口径与四档映射

| 现有项目口径 | 默认映射 | 使用规则 |
|---|---|---|
| `本地核算结果` | 第一档 | 仅当存在完整公式链、脚本结果或独立核算链时成立 |
| `权威资料支撑` | 第二档 | 需保留 URL、出处性质和适用边界 |
| `工程估算` | 第三档 或 第四档 | 只有写明迁移依据时可视为第三档；否则只能降为第四档 |
| `情景参数` | 第四档 | 仅能作为情景开关、灵敏度分析或边界占位，不能冒充已证实输入 |
| `待验证缺口` | 非证据 | 表示当前不能下结论，必须显式保留，不得硬转成参数或结论 |

### 3.3 绝对禁止事项

- 绝对禁止无支撑压入参数。
- 绝对禁止把第三档、第四档证据写成“已证实事实”。
- 绝对禁止把 `待验证缺口` 偷换成“默认合理”。
- 绝对禁止用“行业常识”“通常如此”“大概如此”代替 URL、公式链或推断理由。

## 四、证据采信闸门

### 4.1 AskUserQuestion 触发规则

- 对第二档、第三档、第四档证据，**首次引用即必须 AskUserQuestion 与用户对齐**，不得先斩后奏。
- 这里的“首次引用”包括但不限于：
  - 首次在分析链中把该资料当作支持性证据写入
  - 首次将该资料转化为模型输入、参数边界或比较分母
  - 首次准备在计划、锚点、报告、结论中使用该资料
- AskUserQuestion 不能给用户填空题，必须构造成选择题，至少包含：
  - 是否采信
  - 若采信，按哪一档采信
  - 是否允许进入主链 / 并行分支 / 仅保留为待验证项

### 4.2 无需额外请示的情形

- 第一档证据中的确定性物理推导、本地脚本核算、单位换算、已闭合公式复算，可直接推进。
- 但若第一档推导依赖一个尚未确认的外部输入边界，则该外部输入本身仍需按第二档或更低档触发请示。

### 4.3 证据降级规则

- 若号称第二档的资料缺少 URL、来源权威性或直接适用性说明，自动降级，不得继续按第二档使用。
- 若号称第三档的迁移资料缺少迁移理由或三档预估依据，自动降为第四档或直接转为 `待验证缺口`。
- 若第四档推断没有明确理由、边界与临时用途，直接判为违规，不得入模。

## 五、调研执行顺序

### 5.1 本地建模与外部搜索的平衡

- 原则不是“只搜外部资料”或“只做本地计算”，而是按证据强度排序：
  - 先判断能否直接物理推导或本地建模
  - 再寻找权威外部资料锁定输入边界
  - 再考虑可迁移证据
  - 最后才允许合理推断
- 任何时候，一旦某问题具备直接物理推导链，就必须优先启动本地 Python/公式核算，不得跳过这一步直接引用二手说法。
- 外部搜索必须充分，但搜索的作用是补强与校验，不是替代可直接成立的本地推导。

### 5.2 并行策略

- 对复杂问题优先使用 `parallel-sub-agent` 并行展开，至少优先考虑：
  - 一条分支做外部搜证与 URL 收集
  - 一条分支做本地物理推导 / Python 建模 / 单位校核
- 并行结果合流时，必须以证据等级高者优先，不得因“搜到了更多页面”而压过本地可复核物理链。
- 若并行分支之间出现口径冲突、强弱证据混写或无法裁决的迁移争议，必须回到 AskUserQuestion，而不是由执行者主观拍板。

## 六、写作与锚点规则

### 6.1 计划阶段与锚点写作阶段

- 在计划、spec、note、`.trae/documents/`、根目录研究稿等锚点写作中，严厉禁止主观臆测、倾向性判定与偷换口径。
- 这些文档必须以客观审计视角写作，显式区分：
  - 已证实内容
  - 可复核本地核算结果
  - 允许采信的外部证据
  - 仅能并列保留的工程场景
  - 当前不能下结论的缺口
- 当证据强度不足以支持“翻转”“证明”“已定案”等措辞时，必须降格表述，不得超写。

### 6.2 留痕要求

- 第一档证据应至少落盘以下之一：脚本、公式链、计算表、关键中间量。
- 第二档、第三档、第四档证据一律要求保留来源 URL；缺少 URL 的外部资料不得进入正式产物。
- 第三档、第四档一律要求保留“为什么可用 / 为什么暂时只能这样用”的说明。
- 若某脚本本轮未现场重跑，只能写“已存在落盘结果并完成静态核对”，不得写成“本轮已重跑验证通过”。

## 七、输出风格约束

- 输出应优先给出证据链和约束链，而不是先给判断立场。
- 结论必须与证据强度匹配：强证据可给强结论，弱证据只能给弱表述。
- 同一段落内不得混合使用不同证据等级却不加区分。
- 所有关键数字都必须能回链到 URL、脚本、公式或正式锚点文件。

## 八、执行底线

- 只要是可本地建模独立核算的，均应本地建模独立 Python 核算，并以核算结果为主依据。
- 即使使用外部资料，也必须验证其真实性、准确性、可靠性与迁移边界。
- 在执行过程中，只要出现非确定性、非共识、支撑弱、存在选择、证据冲突或无法客观裁决的情况，必须 AskUserQuestion 与用户对齐。
- 若当前没有足够支撑，应明确输出“当前不可判定”或“待验证缺口”，这属于合规状态，不得硬凑结论。
